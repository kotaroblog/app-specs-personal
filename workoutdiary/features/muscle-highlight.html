筋肉部位ハイライト機能

種目を選択すると、鍛えられる筋肉部位を人体図上にリアルタイムでハイライト表示する機能

planning 機能ID: WD-FEAT-MUSCLE-HIGHLIGHT

概要

ユーザーが種目を選ぶと、その種目で鍛えられる筋肉部位を人体図上でハイライト表示する機能。主働筋・補助筋・安定筋の3段階で関与度を色分けし、トレーニングの効果を視覚的に伝える。ブラウザプロトタイプ（/Users/happyboy1002/workspace/workout-bodymap-proto/）での検証が完了しており、本仕様書はその結果をまとめてiOS本実装の指針とする。

目的・ユーザー価値

  • 種目に対する筋肉への効き方を視覚的に伝え、ユーザーのトレーニング意識を高める
  • 初心者が「どの種目で何が鍛えられるか」を直感的に把握できるようにする
  • 種目選択時の納得感を高め、アプリへの継続エンゲージメントを促進する

競合状況

  • 海外勢（Muscle Booster・Fitbod・Strong）には実装あり
  • 国内筋トレアプリではほぼ未実装 → 国内市場での差別化要素

想定配置

  • 種目詳細画面（種目名タップ時）
  • または種目選択時のサイドパネル / モーダルシート

画面構成

エリア内容
上部選択中の種目名 + 主な効く部位のサマリーテキスト
中央人体図（前面・背面の2ビュー横並び）
左種目リスト（カテゴリ: 胸／肩／背中／腕／腹／脚）
右凡例（主働筋・補助筋・安定筋）＋ 部位一覧

レイアウト分岐（iOS）

  • iPad / 横向き iPhone: 左に種目リスト・中央に人体図・右に凡例+部位一覧を並べる3カラム構成
  • 縦向き iPhone: 人体図メイン表示 + ボトムシートに種目リストを格納するパターン

人体図の描画方針

SVG（Webプロトタイプ）または SwiftUI Path（iOS本実装）で5層構造を採用。「教科書イラスト風」をターゲットとし、写真リアル（皮膚色・血管描写）は避ける。

レイヤー役割実装
Layer 1（ベース） 各筋肉の輪郭 Path。fill 色で関与度を表現 SwiftUI Path / SVG <path>
Layer 2（陰影） radialGradient で周辺減光。立体感を付加 CSS mix-blend-mode: multiply / SwiftUI .blendMode(.multiply)
Layer 3（ハイライト） 中央が明るくなるグラデーション CSS mix-blend-mode: screen / SwiftUI .blendMode(.screen)
Layer 4（筋繊維 Striations） 各筋肉の繊維方向に細い線を引き、解剖図感を出す SVG <line> / SwiftUI Path.move + addLine
Layer 5（補助線） 鎖骨・腹白線・脊柱ラインなど解剖学的サインライン SVG <path>（stroke のみ）

インタラクション

操作動作対応バージョン
種目をタップ 該当筋肉が 0.25s ease のカラーアニメーションでハイライト。非関与筋はグレーに戻る MVP

iOS 実装方針

3つの実装案を検討。推奨は第1案（SwiftUI Path）。

案方針メリットデメリット
第1案（推奨） SwiftUI Path + Canvas でネイティブ描画。プロトのSVGパスをパースして Path に変換するヘルパーを実装 SwiftUI エコシステムに完全統合。アニメーションが自然。ダークモード対応容易 SVG → Path 変換の初期工数が必要
第2案 SVGファイルを Asset として読み込み、SVGKit（OSSライブラリ）で描画 SVGをそのまま使える。変換工数ゼロ 外部依存。ライブラリのメンテ状況に依存。細かい制御が難しい
第3案 HTML/SVG を WKWebView で表示。プロトコードをほぼそのまま流用 実装最速。プロトの検証資産を直接利用可 ネイティブUIとの統一感に課題。メモリ・パフォーマンスのオーバーヘッド

Swiftデータモデル

enum Intensity: String, Codable {
    case primary     // 主働筋
    case secondary   // 補助筋
    case stabilizer  // 安定筋
}

enum MuscleSide: String, Codable {
    case front, back
}

struct Muscle: Identifiable {
    let id: String          // 例: "chest-upper"
    let label: String       // 例: "大胸筋（上部）"
    let side: MuscleSide
}

struct MuscleActivation: Identifiable {
    let id: String          // Muscle.id
    let intensity: Intensity
}

struct ExerciseMuscleData: Identifiable {
    let id: String          // 例: "bench-press"
    let name: String        // 例: "ベンチプレス"
    let category: String    // 例: "胸"
    let activations: [MuscleActivation]
}

MVP / v2 / v3 段階定義

フェーズ内容対応種目・部位数
MVP 28種目 × 22部位の静的データで人体図ハイライト。種目選択 → ハイライト表示のみ。逆引き・ヒートマップなし 28種目 / 22部位
v2 筋肉タップで関連種目を逆引き表示。種目登録時に該当筋肉が自動ハイライト。iPad ホバーツールチップ 同上（逆引き機能追加）
v3 過去1週間 / 1ヶ月の累積負荷を筋肉ヒートマップで表示。自分のトレーニング偏りを可視化する「筋肉バランスチェック」 実績データと連動

ビジネスルール

  • 筋肉データ（22部位 ID・ラベル）はコードバンドル内の静的 JSON または Swift 定数として保持する。ネットワーク通信不要
  • 種目データ（28種目の筋肉マッピング）も同様に静的データとして管理。将来的にはユーザー種目登録時にカスタムマッピングを付与可能とする
  • 種目リストの表示順: カテゴリ（胸→肩→背中→腕→腹→脚）→ 各カテゴリ内はアルファベット/名称順
  • 1種目に紐付けられる筋肉は任意数（上限なし）。ただし主働筋（primary）は必ず1つ以上
  • 同一筋肉に複数の関与度を付けることは禁止（1部位につき intensity は1つ）
  • v3 累積負荷ヒートマップは Premium 機能として提供予定

外部連携

連携先用途備考
なし（静的データのみ）MVP はネットワーク通信なしv3 でサーバー側累積計算が必要になる場合あり
SwiftData（v3）WorkoutSet の累積負荷集計v3 実装時に既存クエリを拡張

エラー処理

発生条件対応
選択された種目IDに対する筋肉マッピングが存在しない 全筋肉をグレー（非関与）のまま表示。エラーは Logger で warning レベルに記録
SVG / Path データが不正で描画に失敗 人体図エリアに「表示できません」プレースホルダーを表示。LogLevel: error で記録
アニメーション中に種目を連続タップ 直前のアニメーションをキャンセルして最新の選択状態を即時反映

実装メモ

  • プロトタイプのSVGパス座標をそのままSwiftUI Pathに変換する SVGPathParser ヘルパーを先行実装すると工数削減になる
  • SwiftUI withAnimation(.easeInOut(duration: 0.25)) で色変更アニメーションを実装。Color は直接アニメーション対象になるため追加対応不要
  • 前面・背面の2ビューは HStack または TabView(style: .page) で切り替える。縦向きiPhoneでは TabView 推奨
  • 筋肉部位の Path はプロトの bodymap.js 内の SVG path 文字列から変換。変換スクリプトは Node.js で一度だけ実行してSwiftファイルを生成する方法が最速
  • 将来の v3 ヒートマップでは同じ MuscleHighlightView に累積負荷 Double（0.0〜1.0）を渡す設計にしておくと拡張が容易
  • ダークモード対応: ハイライト色は変えず、ベースカラー（非関与グレー）とアウトラインだけ Asset Catalog でモード分岐
  • #Preview マクロでサンプル種目（ベンチプレス）を選択状態にした状態を必ず記述
  • print() デバッグ禁止。Logger(subsystem: "WorkoutDiary", category: "MuscleHighlight") を使用
