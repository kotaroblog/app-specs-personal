データモデル

SwiftData @Model の構造とリレーション

ステータス: 企画確定・実装未着手（2027-Q1リリース目標）。本書は設計方針であり、以下は「〜する設計とする」前提で読むこと。新規4モデル＋旧PokerLesson流用2モデルの計6モデルを Schema([Session, Venue, Hand, WeeklyReview, Question, AnswerRecord]) で構成する設計とする。

ER 図

erDiagram Venue ||--o{ Session : has Session ||--o{ Hand : has Question ||--o{ AnswerRecord : has Venue { UUID id String name Date createdAt } Session { UUID id Date date Int buyIn Int cashOut Int durationMinutes String gameTypeRaw String stakes String note Date createdAt } Hand { UUID id String entryModeRaw String holeCard1 String holeCard2 StringArray boardCards String positionRaw String scenarioRaw String actionRaw String resultRaw StringArray weaknessTagsRaw String note Date createdAt } WeeklyReview { UUID id Date weekStartDate Int profitSnapshot Int sessionCount Int handCount StringArray topWeaknessTagsRaw String nextFocus Date completedAt Date createdAt } Question { UUID id String positionRaw String scenarioRaw String handStr Int heroStack String correctActionRaw Bool isFreeTrial } AnswerRecord { UUID id String selectedActionRaw Bool isCorrect Date answeredAt }
ER図中の StringArray は mermaid のパース制約回避のための表記であり、実際の型は [String]（boardCards / weaknessTagsRaw / topWeaknessTagsRaw）。durationMinutes / completedAt は実際には Optional（Int? / Date?）。
リレーション補足: Hand.session は任意（クイックメモは未紐づけで作成可）。WeeklyReview はスタンドアロン（他モデルと直接のリレーションを持たない）。

新規モデル

Session

@Model
final class Session {
    var id: UUID = UUID()
    var date: Date = Date()              // プレイ日
    var buyIn: Int = 0                   // 円
    var cashOut: Int = 0                 // 円
    var durationMinutes: Int? = nil      // 任意（Pro時給分析用）
    var gameTypeRaw: String = "tournament" // tournament / ring
    var stakes: String = ""              // リング用ブラインド表記（任意）
    var note: String = ""
    var createdAt: Date = Date()

    var venue: Venue? = nil
    @Relationship(deleteRule: .nullify, inverse: \Hand.session)
    var hands: [Hand] = []

    var profit: Int { cashOut - buyIn }  // computed
}

Venue

@Model
final class Venue {
    var id: UUID = UUID()
    var name: String = ""
    var createdAt: Date = Date()
    @Relationship(deleteRule: .nullify, inverse: \Session.venue)
    var sessions: [Session] = []
    // 店別通算成績（訪問数・累計損益）はクエリ／computedで算出。非正規化しない
}

Hand

@Model
final class Hand {
    var id: UUID = UUID()
    var entryModeRaw: String = "quick"   // quick（5秒メモ）/ simple（30秒簡易記録）
    var holeCard1: String = ""           // "Ah" 形式（Rank+Suit rawValue）
    var holeCard2: String = ""
    var boardCards: [String] = []        // 0〜5枚
    var positionRaw: String = ""         // PokerPosition rawValue（9max）
    var scenarioRaw: String = ""         // rfi / vs_open / cold_call_vs_squeeze / other
    var actionRaw: String = ""           // fold / call / raise（自分のプリフロップ選択）
    var resultRaw: String = ""           // won / lost / chopped / unknown
    var weaknessTagsRaw: [String] = []   // WeaknessTag rawValue配列
    var note: String = ""                // 1行メモ
    var createdAt: Date = Date()

    var session: Session? = nil          // 任意。クイックメモは後からセッションに紐づけ可
}

WeeklyReview

@Model
final class WeeklyReview {
    var id: UUID = UUID()
    var weekStartDate: Date = Date()     // 週の月曜0時
    var profitSnapshot: Int = 0          // 完了時点の週間収支
    var sessionCount: Int = 0
    var handCount: Int = 0
    var topWeaknessTagsRaw: [String] = [] // 頻出タグ上位3
    var nextFocus: String = ""           // 「次回の課題」（ホームに表示する核）
    var completedAt: Date? = nil         // nil=未完了
    var createdAt: Date = Date()
}

設計判断

  1. クイックメモは独立モデルにしない — Hand に entryModeRaw（quick / simple）で統合する。「後から詳細化」は同一レコードの編集で実現し、履歴一覧・セッション紐づけを一本化する。
  2. カードは "Ah" 形式の String 2フィールド — 旧PokerLesson の PokerCard（Rank/Suit enum）・HandNotation と相互変換する。ボードのみ [String]（0〜5枚可変）とする。
  3. 弱点タグは enum rawValue 配列 — マスタ @Model 化しない。v1.0 はプリセット固定・集計は取得後に in-memory で実施する（個人記録の規模で十分）。プリセット案10種: レンジ外参加／ポジション軽視／コール過多（降りられない）／オーバーブラフ／バリュー逃し／ベットサイズ不適切／ティルト／スタック管理ミス／相手レンジ決め打ち／ドロー追いすぎ（実装時に確定）。
  4. MTT照合結果（HeroPreflopResult）は保存しない — positionRaw ＋ ホールカード ＋ scenarioRaw から流用 RangeDataService で都度再計算する（正規化・データ最小主義）。
  5. 旧 UserProgress（ストリーク）は持ち込まない — why_return 設計でストリーク・ノルマ型は不採用のため。答え合わせの成績は AnswerRecord から都度算出する。
  6. VersionedSchema を初回から整備する — 旧PokerLesson には無かった。PokerDiarySchemaV1 ＋ SchemaMigrationPlan を v1.0 で導入する（MVP #14）。
  7. 金額は Int（円） — profit は computed。4+維持のため画面文言は「収支」で統一し、賭博表現を使わない。

マイグレーション方針

SwiftData の @Model は全プロパティにデフォルト値必須（CLAUDE.md準拠）。上記6モデルはすべてデフォルト値を設定済み。新規プロパティ追加時も必ずデフォルト値を設定する。

ModelContainer 生成失敗時は in-memory フォールバック ＋ Logger.error で継続する設計とする（旧PokerLesson の方式踏襲）。
