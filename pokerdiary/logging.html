ロギング方針

os.log / Logger を用いたログ出力ルール

ステータス: 企画確定・実装未着手（2027-Q1リリース目標）。本書は設計方針であり、以下は「〜する設計とする」前提で読むこと。

基本方針

print() は禁止。Logger（os.log）を使用する（CLAUDE.md準拠）。

subsystem は Bundle.main.bundleIdentifier（＝ com.happyboy1002.PokerDiary）を用いる。
段階1は os.log 出力のみとし、リリース後に Firebase Analytics 等のラッパーへ差し替え可能な設計とする（旧PokerLesson EventLogger の方式踏襲）。

Logger カテゴリの定義

旧PokerLesson の「ファイル単位カテゴリ」から「役割単位カテゴリ」に整理する方針とし、8カテゴリを定義する。

import OSLog

extension Logger {
    private static let subsystem = Bundle.main.bundleIdentifier ?? "com.happyboy1002.PokerDiary"

    static let app = Logger(subsystem: subsystem, category: "app")               // 起動・ライフサイクル・コンテナ生成
    static let data = Logger(subsystem: subsystem, category: "data")             // SwiftData保存・取得・マイグレーション
    static let kpi = Logger(subsystem: subsystem, category: "kpi")               // EventLogger（KPI専用）
    static let purchase = Logger(subsystem: subsystem, category: "purchase")     // SubscriptionManager（StoreKit 2）
    static let ad = Logger(subsystem: subsystem, category: "ad")                 // AdMobバナー
    static let notification = Logger(subsystem: subsystem, category: "notification") // 週次レビュー通知
    static let range = Logger(subsystem: subsystem, category: "range")           // レンジ表・130問データロード（流用系）
    static let share = Logger(subsystem: subsystem, category: "share")           // シェアカードImageRenderer
}

ログレベルの使い分け

レベル用途PokerDiary での例
.debug開発時のみ確認したい情報画面遷移・VM初期化
.info通常のイベントセッション保存完了・照合実行
.notice注目すべきイベント初回起動・サブスク状態変化
.warning異常だが継続可能通知許可拒否・広告ロード失敗
.errorエラー、機能継続不可SwiftData保存失敗・レンジJSONロード失敗
.criticalアプリ継続不可ModelContainer生成失敗（in-memoryフォールバック発動）

プライバシー対応（ファイナンスアプリとして最重要）

収支金額（buyIn / cashOut / profit）・店名・メモ本文・次回の課題テキストはログに出さない。 やむを得ない場合は privacy: .private を必須とする。

KPIイベントのパラメータに金額・店名・自由記述を絶対に含めない（件数・真偽値・enum値のみ）。 
// OK（ID を .public で出力。金額・店名を含まない）
Logger.data.info("セッション保存完了 id=\(session.id, privacy: .public)")

// NG（収支金額と店名を公開ログに出している）
Logger.data.info("収支: \(session.profit)円 @\(venue.name)")

KPIイベント設計（EventLogger）

旧PokerLesson の EventLogger（os.log出力・DEBUGビルドのみメモリ内リングバッファ最大100件＋設定画面の EventLogDebugView）を流用し、AppEvent enum を PokerDiary 用に新設計する。

イベント名パラメータ計測目的
onboarding_completedなし初回体験の到達率
session_loggedgame_type, has_duration, venue_is_new錨行動の頻度（最重要リテンション指標）
hand_prompt_shownなし完了画面「1手残す?」表示数
hand_prompt_acceptedなし同・承諾数（収支→ハンドの最重要ファネル）
hand_loggedentry_mode, tag_count, has_session差別化行動の量と質
draft_link_prompt_shownなし下書き紐づけ提案の表示
draft_link_acceptedなし同・承諾。クイックメモ→セッション接続のファネル
preflop_match_shownmatched照合バナー表示・GTO一致率
range_viewer_openedsource従機能の受動利用
drill_answeredcorrect答え合わせ利用
weekly_review_startedsourceレビュー着手（notification / home 別）
weekly_review_completedtag_count, has_next_focusレビュー完遂（週次リテンションの核）
share_card_createdなしGTM素材生成
share_card_sharedなし実共有（ストア外流入の種）
paywall_shownsource（7種）課金ファネル入口
subscribe_completedplan課金成立
ファネル設計思想: 「abandoned」イベントは直接出さず、shown と accepted / completed の突合で離脱率を算出する（旧PokerLesson方式踏襲）。

全イベントは privacy: .public で出力可能な値のみで構成する。金額・自由記述を含まない設計だから成立する。

禁止事項

  • print() の使用（CI / lint で検知）
  • APIキー・パスワード・課金トランザクション詳細の出力
  • 収支金額・店名・メモ・次回の課題テキストを公開ログに含めること

確認方法

macOS の Console.app でデバイスを選択し、subsystem（com.happyboy1002.PokerDiary）でフィルタすると当該アプリのログのみ確認可能。
