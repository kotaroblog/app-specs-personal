主要新機能: パネル数選択

概要

従来 2×2 固定（4パネル）だったゲーム盤面を、プレイヤーが 2 / 4 / 6 / 8 から選べるように拡張。 難易度モード（NORMAL / FAST / INSANE）と独立したカスタマイズ軸として機能する。 パネル数が増えると記憶すべき色の組み合わせパターンが増加し、視覚的な難易度が上がる。

PanelCount enum

enum PanelCount: Int, CaseIterable, Identifiable {
    case two = 2     // 1×2 グリッド
    case four = 4    // 2×2 グリッド（v1.0 既存仕様）
    case six = 6     // 2×3 グリッド
    case eight = 8   // 2×4 グリッド
}

UI 変更（GameView）

  • GameView に PanelCount セレクター（ピル型ボタン）を追加
  • セレクター配置: 難易度 Picker の近接位置（idle 状態でのみ操作可能）
  • パネル数変更後はゲームリセット → 新パネル数でラウンド開始
  • プレイ中はセレクターを無効化（誤操作防止）

ベストスコア管理

ストレージキー体系

パネル数×難易度の組み合わせごとに独立したベストスコアを記録する。

キーパターン例説明
chromatic_best_score_p{N}_{speed}chromatic_best_score_p4_normalパネル数4・NORMAL のベスト
chromatic_best_score_p4_fastパネル数4・FAST のベスト
chromatic_best_score_p4_insaneパネル数4・INSANE のベスト
chromatic_best_score_p8_insaneパネル数8・INSANE のベスト

N は 2 / 4 / 6 / 8、{speed} は normal / fast / insane の 3 値。合計 12 エントリ。

旧キーからのマイグレーション

旧キー（パネル数なし）移行先処理タイミング
chromatic_best_score_normalchromatic_best_score_p4_normal初回起動時に自動マイグレーション
chromatic_best_score_fastchromatic_best_score_p4_fast同上
chromatic_best_score_insanechromatic_best_score_p4_insane同上

マイグレーション済みフラグ: chromatic_best_score_migrated_v1_2（Bool）。二重マイグレーション防止。

注意: 旧キーのデータは上書きせず、移行後も残す（後方互換）。旧キーへのアクセスは削除しない。

Game Center リーダーボード

Game Center へのスコア送信は パネル数 4（2×2）のみ に限定する。 v1.0 時点のリーダーボード（normal / fast / insane 各 1 本、計 3 本）との整合性を維持するため、 パネル数 2 / 6 / 8 のスコアはローカルのみ保存し、リーダーボードには反映しない。

パネル数Game Center 送信理由
2なし既存リーダーボードとスコア互換性なし
4あり（3難易度）v1.0 既存仕様を継承
6なし同上
8なし同上

将来的にパネル数別リーダーボードを追加する場合は、App Store Connect での新規リーダーボード登録が必要。

ビジネスルール

  • パネル数はゲーム開始前（idle 状態）にのみ変更可能。プレイ中は変更不可
  • パネル数を変更するとゲームは即座にリセットされる（スコア消失なし）
  • ベストスコアはパネル数×難易度の組み合わせ（12 通り）ごとに独立管理
  • Game Center へのスコア送信はパネル数 4 のみ（既存リーダーボードとの互換性維持）
  • Daily Challenge はパネル数設定に依存しない独立モード（常に 4 パネル固定）
  • 旧ユーザーのベストスコアはパネル数 4 として自動マイグレーション（二重実行防止あり）

エラー処理

発生条件対応
マイグレーション処理中のクラッシュchromatic_best_score_migrated_v1_2 フラグが false のまま残るため次回起動時に再試行。べき等な処理なので問題なし
PanelCount の不正値が UserDefaults に保存されていた場合デフォルト値（4）にフォールバックして起動継続
パネル数変更時のグリッドレイアウト崩れresetGame() 後に再描画するため発生しない。ただし動的に変更した場合は ForEach の id 変更で強制再構築
