SceneLabelLayout 単一ソース化（2026-06-10）

撮影プレビュー（ShootingView / SingleStepShootingView）と完成動画（VideoCompositionService）の本編テロップを、配置比率・サイズ比率・行折り返し挙動・descent slack 補正まで含めて単一の SceneLabelLayout enum に集約。プレビューと書き出しの WYSIWYG を達成。

  • 実装ファイルViews/Common/SceneLabelLayout.swift
  • 配置比率leadingRatio = 0.05 / bottomRatio = 0.22 / maxWidthRatio = 0.55 / lineSpacingRatio = 0.008
  • サイズ比率stepNameFontSizeRatio = 0.034 / timeFontSizeRatio = 0.024
  • メトリクス補正lineHeightFactor = 1.17 / textHeightFactor = 1.4 / descentSlackFactor ≒ 0.23
  • 行数上限maxLines = 2（プレビュー / 動画書き出し共通）
  • SwiftUI 用ヘルパleadingPadding / bottomPadding / maxWidth / lineSpacing
  • CALayer 用ヘルパlayerOriginX / layerOriginY(fontSize:) / layerTextWidth / layerLineSpacing

撮影プレビュー位置を 22% に統一

  • 旧cardHeight * 7%（プレビューと動画でズレ）
  • 新SceneLabelLayout.bottomPadding(cardHeight:) = cardHeight * 22%
  • 効果完成動画の本編テロップ位置（renderHeight * 22%）と一致 → WYSIWYG 達成

長文ステップ名の折り返し挙動を統一

  • SwiftUI Text.lineLimit(SceneLabelLayout.maxLines) + .truncationMode(.tail)
  • CATextLayerisWrapped = true + truncationMode = .end + frame.height = singleLineHeight × actualLineCount
  • 行数測定actualStepNameLineCount(text:renderSize:fontName:) で NSAttributedString.boundingRect により実行数測定（最大 maxLines にクランプ）

descent slack 補正

  • 課題CATextLayer は top-anchored 描画でグリフ下端が frame 下端より約 fontSize * 0.23 浮く
  • 対応layerOriginY(fontSize:) は base - fontSize * descentSlackFactor で下方向に補正
  • 効果SwiftUI Text の intrinsic 高さと概ね揃い、プレビュー / 動画の見え方がほぼ一致

SingleStepShootingView の追従

  • 底面グラデーション追加ShootingView と同じグラデーションを敷設（22% 化でテロップが暗部に乗る副作用の対応）
  • #Preview マクロ規約準拠で追加

  • 中央: ProgressView

  • 配置トップバー右（HStack trailing）
  • 表示条件.ready 状態のみ（録画中・カウントダウン中は非表示）
  • 非表示時プレースホルダ録画中・カウントダウン中に反転ボタンを非表示にする際、44pt の透明プレースホルダを配置（旧: 36pt）。タイトルの横ズレを防止しトップバーのレイアウトを安定させる
  • 変更前下部コントロール左側で「撮り直し」ボタンと排他表示。1本撮影後は反転不可だった

SingleStepShootingView 機能パリティ（4471820）

単発撮影（SingleStepShootingView）に、連続撮影（ShootingView）が v1.1.0 で実装済みだった2機能を横展開し、操作方法を統一した。

カメラ反転ボタン（トップバー）

  • 配置トップバー右（ShootingView と同じ位置）
  • 表示条件撮影前（録画中・カウントダウン中・保存中でない状態）のみ表示
  • 変更前SingleStepShootingView にカメラ反転機能自体が存在しなかった
  • 変更後ShootingView と同じ条件・位置でカメラ反転が可能になり、前面・背面カメラを自由に切り替えられる

カウントダウン中タップでスキップ

  • 動作カウントダウン中に録画ボタンをタップすると skipCountdown() が呼ばれ即座に録画開始（ShootingView と同仕様）
  • 変更前カウントダウン中は録画ボタンが完全に無効化されており、スキップ不可だった
  • 変更後録画ボタンのタップ判定をカウントダウン中も有効化し、待ちのストレスを排除

録画中タップで早期停止・保存

  • 動作単発撮影（SingleStepShootingView）の録画中にタップすると録画を即時停止してクリップを保存する。連続撮影（ShootingView）と挙動を統一
  • 変更前単発撮影では録画時間が経過するまで停止できず、撮り直しには一度タイムアップを待つ必要があった
  • 変更後録画ボタンが録画中は白い停止四角（stop.fill）に切り替わり、タップで早期停止・保存が可能

機能パリティ確認: これにより連続撮影と単発撮影でカメラ反転・カウントダウンスキップ・録画早期停止の操作が統一された。単発撮影はクリップライブラリの各ステップ行「撮影/再撮影」ボタン、またはClipGenerateViewの空ステップ「撮影する」から起動する。

連続撮影の中断 → クリップ保持（v1.1.0・#1）

v1.1.0 以前は連続撮影を中断した際、録画済みのクリップもすべて破棄されていた。v1.1.0 でクリップ保持に変更し、中断コストを大幅に軽減した。

  • 変更前中断でセッション内の全クリップを破棄
  • 変更後録画済みクリップはクリップライブラリに保持。録画中の現在ステップのクリップのみ破棄
  • 制御フラグisCancelled フラグで録画中ステップのみ識別して破棄対象にする
  • 中断アラート「撮影を中断しますか？」→「保存して終了」（録画済みクリップを保持して閉じる）/ 「キャンセル」（撮影に戻る）
  • アラートトリガー閉じる × ボタンタップ（ShootingView / SingleStepShootingView 共通）

Dynamic Type クランプ（v1.1.0・#5）

固定サイズフォントが多いため、アクセシビリティ設定で過大なサイズを選択した場合のレイアウト崩れを防止するクランプを設定した。

  • 設定箇所RootView（アプリ全体に適用）
  • 制限範囲.dynamicTypeSize(.xSmall ... .accessibility2)
  • 上限の根拠accessibility3 以上は固定フォントを多用する撮影 UI / クリップサムネ等でレイアウト崩れが確認されたため
  • v1.2 方針セマンティックフォント（.headline / .body 等）へ段階的に移行してクランプを外す方向

撮影ハプティクス（v1.1.0）

スマホを置いて撮影するユースケースで、画面を見ずに録画状態を把握できるよう .sensoryFeedback による触覚フィードバックを追加した。録画音を鳴らすと動画音声トラックに混入するため、ハプティクスのみで通知する。

タイミングフィードバック種別説明
録画開始.start録画が開始されたことを触覚で通知
録画完了（クリップ保存）.successクリップが正常に保存されたことを通知
カウントダウン（1秒ごと）.selectionカウントダウン中の各ティックを触覚で通知
  • 実装.sensoryFeedback(_:trigger:) モディファイアを ShootingView・SingleStepShootingView の両方に適用
  • 録画音なしシステム音は動画音声トラックに混入するため廃止。ハプティクスのみで代替
  • 対象ShootingView（連続撮影）/ SingleStepShootingView（単発撮影）の両方

撮影中断ダイアログの3択化（v1.2）

v1.1.0 の「保存して終了 / キャンセル」2択から、制作摩擦をさらに低減するため3択に変更した。

選択肢動作
クリップを残して終了録画済みクリップをライブラリに保持してセッションを閉じる（旧「保存して終了」相当）
破棄して終了そのセッションの全クリップを削除してセッションを閉じる。DB レコード確定後にファイルを削除する順序で参照整合性を保証（「DBが消えたのにファイルが残る」孤立ファイルを防止）
続けるダイアログを閉じて撮影を再開

「破棄して終了」実行時の削除順序: modelContext.delete(clips) → modelContext.save() → ファイル削除（ClipStorageService.deleteClip）。DB を先に確定することで、ファイル削除の途中でアプリが強制終了しても孤立した DB レコードが残らない。

続きから撮るモード（skipStepsWithExistingClips）（v1.2）

ShootingViewModel に skipStepsWithExistingClips モードを追加し、既に撮影済みのステップをスキップして未撮影ステップのみを連続撮影できるようにした。

  • 起動方法ClipLibraryView の「残りのNステップを続けて撮影」CTA、またはホーム画面の「撮影の続き」カードから startMode: .resumeShooting で起動
  • ステップ選択Routine.unshotSteps（モデルの computed property）を使って未撮影ステップのみを対象に連続撮影を開始
  • 完了後の遷移プレビュー画面直行ではなく、クリップ選択画面（最新クリップが自動選択済み）に遷移。既存ステップ分のクリップも含めて動画を生成できる
  • 判定ソースRoutine.unshotSteps / Routine.isPartiallyShot をモデルに集約し、複数画面で共有（単一の判定ソース）
  • 関連続きから撮る導線の機能仕様書

撮影ファネル計測（v1.2）

撮影セッション開始・中断・完了の3イベントを追加し、撮影完遂率の可視化と離脱ポイントの特定を可能にした。

イベント発火タイミングパラメータ
capture_startedセッション内の初回録画開始時（1セッション1回）total_steps: Int
capture_interrupted撮影画面をユーザーが途中離脱（中断ダイアログで「クリップを残して終了」or「破棄して終了」を選択）recorded_clips: Int, total_steps: Int

recorded_clips で「何ステップ目で離脱するか」を可視化し、制作摩擦の改善対象を特定する。詳細は Analytics 計測基盤 参照。

テロップ・カメラ品質改善（e216de7, e012cc3, e3e6ca1）

テロップ時刻を録画開始時刻に統一（e012cc3）

従来は録画完了時刻をテロップに焼き込んでいたため、ステップ撮影が分をまたぐと撮影プレビュー（開始時刻表示）と完成動画（完了時刻）が食い違っていた。録画開始時刻に統一することで WYSIWYG を回復した。

  • 変更前録画完了（persistClip 呼び出し）時の Date.now をテロップ時刻として採用
  • 変更後録画開始時に recordingStartedAt = Date.now をキャプチャし、persistClip に渡してテロップ時刻として使用
  • 効果撮影プレビューの時刻表示（開始時刻）と完成動画のテロップ時刻（開始時刻）が一致。分またぎの食い違いを解消

撮影中テロップが選択フォントを反映（e216de7）

Pro の「フォント全解放」で選択したフォントが、撮影プレビューの SceneLabel に反映されるようになった。以前は撮影プレビューが常にデフォルトフォントで表示されており、完成動画（選択フォント反映済み）と一致しなかった。

  • 変更前SceneLabel が HelveticaNeue-Light 固定（ルーティンの選択フォントを参照していなかった）
  • 変更後ShootingViewModel が routine.fontFamily を参照し、SceneLabel の fontName を動的に設定
  • 非Premium時Pro 限定フォントが選択されていてもサブスクなし時はデフォルトフォント（HelveticaNeue-Light）にフォールバック
  • 効果撮影プレビューと完成動画でフォントが一致（WYSIWYG 完全一致）

フロントカメラのミラー録画（e216de7）

自撮り（スキンケア / 朝活）撮影時に、撮影プレビューは鏡像（左右反転）で表示されるが録画ファイルは非反転で保存されるため、完成動画が反転して見える問題があった。録画側の isVideoMirrored を設定することでプレビューと録画ファイルの左右向きを統一した。

  • 変更前フロントカメラでプレビューは鏡像・録画ファイルは非反転 → 完成動画が左右反転して見える
  • 変更後AVCaptureConnection.isVideoMirrored = true（フロントカメラ接続時）で録画ファイルも鏡像として保存 → プレビューと完成動画の左右向きが一致
  • 対象フロントカメラ使用時のみ。バックカメラは変更なし

クリップ生成の時刻フォーマット統一（e012cc3）

連続撮影経路（ShootingView）とクリップ生成経路（SingleStepShootingView / ClipGenerateView）でクリップ保存時の時刻フォーマット（12時間制 / 24時間制）が食い違っていた問題を、AppSettings.timestampFormat を単一参照先として一本化した。

  • 変更前各撮影経路が独自のフォーマットロジックを持っており、同一設定でも経路によって表記が異なった
  • 変更後全経路が AppSettings.shared.timestampFormat を参照。設定変更が即時に全経路へ反映される

「後で撮る」スキップ発生時の撮影完了後ルーティング（52c8a94）

連続撮影中に「後で撮る」（旧「次へ」）を使ってステップをスキップした場合、撮影完了後の遷移先を変更する。

  • hasSkippedSteps フラグShootingViewModel に追加。「後で撮る」操作が1回でも発生すると true に設定。セッション内でリセットしない
  • スキップあり（hasSkippedSteps == true）撮影完了後は ClipGenerateView（クリップ選択画面）に遷移。続きから撮る（resumeShooting）モード完了後と同じ着地点に統一
  • スキップなし（全ステップ撮影済み）従来通りプレビュー画面直行
  • 設計意図スキップで未撮影ステップが残った状態でプレビューに直行すると、未撮影ステップのクリップが欠落した状態で動画生成が行われてしまう。ClipGenerateView 経由にすることで、ユーザーが選択状態を確認・調整してから動画生成できる
  • 関連クリップライブラリ（スキップ着地点としての ClipGenerateView 仕様）/ 続きから撮る導線

SingleStepShootingView 計測パリティ（52c8a94）

単発撮影（SingleStepShootingView）の保存成功時に、連続撮影と同じ firstRecording / streakUpdated イベントに加え、新規イベント singleStepCaptureCompleted を発火するよう変更した。

  • 新規イベントsingleStepCaptureCompleted（保存成功時に発火）
  • 既存イベント（連続撮影と共通）firstRecording（初回のみ）/ streakUpdated（撮影都度）。これらは単発撮影でも以前から発火済み
  • 目的クリップライブラリ経由の単発撮影を Analytics で独立して追跡可能にする。連続撮影の capture_started / capture_completed との対比で計測カバレッジを確保
  • 詳細Analytics 計測基盤 参照

カメラ準備状態（SetupState）（acf8f50）

カメラの起動を CameraService.SetupState として明示的に状態管理し、黒画面のまま録画ボタンを押せてしまう問題を解消した。View はこの1つの状態で「準備中／利用可能／権限拒否／起動失敗」を出し分ける。

状態意味カメラカードの表示
idleセットアップ未着手黒（プレビューなし）
preparing権限確認〜セッション構成〜起動待ちCameraPreparingOverlay（スピナー + 「カメラを準備中…」）。録画ボタン無効化
readyセッション稼働中・録画可能カメラプレビュー。録画ボタン有効
permissionDeniedカメラ権限が拒否されている設定導線アラート
failed権限以外の起動失敗（cameraUnavailable / configurationFailed 等）CameraFailedOverlay（「カメラを起動できませんでした」+ 「再試行」ボタン → CameraService.retry()）
  • 録画ボタンのガードカメラ準備完了（ready）まで録画開始ボタンを無効化し、黒画面での録画開始を防止
  • 世代管理generation トークンで prepare / stop / retry / flip の割り込みを管理。画面を閉じた後や再試行後に古い prepare チェーンが UI 状態を書き換えない構造（retry() はリセット後の世代を引き継ぐ）

進捗ドットの状態を色＋形状で区別（acf8f50）

ステッププログレスバーの各ドットを、色だけでなく塗り／中空（アウトライン）の形状差でも区別するようにした。判定は View から切り離した純粋関数 ShootingProgress.status(for:currentIndex:recordedIndices:skippedIndices:) に集約し、単体テスト可能（ShootingStepStatusTests / ShootingViewModelRecordedIndexTests）。

状態（ShootingStepStatus）形状・色VoiceOver 状態語
currentピンク→オレンジ グラデ（塗り・22×4pt）現在のステップ
recorded濃色（塗り・primaryText 0.85）撮影済み
skipped濃色のアウトライン（中空）＝撮影済みと明確に区別スキップ済み
pending薄グレー（塗り・tertiaryText 0.25）未撮影

変更前: 撮影済みとスキップ済みが同じ表示で区別できなかった。優先順位は current > recorded > skipped > pending（recorded と skipped の矛盾入力でも撮影済みを優先して安定した結果を返す）。

録画ボタン付近の状態説明テキスト（acf8f50）

録画ボタンを押すと何が起こるかを言葉で示す説明テキストを、カメラ映像・SceneLabel を隠さない操作エリア内（録画ボタンの上）に追加した。

状態テキスト
待機中（ready）「○秒録画」（ステップの録画秒数）
カウントダウン中「タップですぐ開始」
録画中「停止して次へ」／最終ステップは「停止して完了」
保存中「クリップを保存中…」

トップバーに現在のステップ名を主表示（acf8f50）

トップバーの主表示を現在のステップ名（16pt medium）に変更し、ルーティン名とステップ番号（N/総数）を従表示（12pt light）に降格した。撮影中に「今どのステップを撮っているか」が一目で分かる。

  • 変更前ルーティン名 + ステップ番号のみ（ステップ名は非表示）
  • 省略挙動ステップ名は lineLimit(1) + truncationMode(.tail) + minimumScaleFactor(0.85) で左右ボタンと重ならないよう1行省略

マイク権限拒否時の非遮断バナー（acf8f50）

マイク権限が拒否されていても撮影をアラートで止めず、映像のみで撮影を続けられることを伝える非遮断バナー MicrophoneDisabledBanner を追加した。

  • 判定CameraService.isMicrophoneDenied（true でも映像のみで録画継続）
  • 文言「マイクが許可されていないため、音声なしで撮影します」（mic.slash.fill アイコン + 黒半透明カプセル）

カメラ反転の連打防止（acf8f50）

  • フラグCameraService.isSwitchingCamera。反転処理中は反転ボタン・録画開始ボタンを無効化し、session への同時操作（多重反転）を防止
  • 切替先の取得反転後のカメラ位置は session.inputs から読み取る（状態の二重管理を回避）
