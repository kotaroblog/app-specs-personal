クリップタブ動線（3分岐・596896d）

旧実装は「クリップタブタップ = routines.first で即撮影開始」だったが、ユーザーの意図（ライブラリを見たい）と乖離していたため修正。

ルーティン件数動作
0件（ルーティン未作成）handleCenterPlusTap() 経由でルーティン作成誘導（PresetSelect または Paywall）
1件その唯一のルーティンの ClipLibraryView を直接起動（startMode: .clipLibrary）
2件以上ClipRoutineSelectSheet（ルーティン選択シート）を sheet で表示し、ユーザーが選んだ後に ClipLibraryView を起動

クリップタブタップで即撮影開始していたバグは 3b93040 で解消（致命3件のうちの1件）。

SessionPayload と StartMode（596896d）

activeSession の型を Routine から SessionPayload（Identifiable struct）へ変更し、起動経路ごとの初期画面を制御する。

  • SessionPayloadlet routine: Routine + let startMode: CreateSessionView.StartMode（UUID で Identifiable）
  • .clipLibraryClipLibraryView をルート表示。マイルーティン行タップ / クリップタブ / ⋯メニュー「クリップライブラリを開く」で使用
  • .shootingSequential起動と同時に連続撮影（ShootingView）を立ち上げる。「次に撮影」CTA / Quick Start チップ / 通知タップ / URL スキームで使用

init 内で _shootingTarget を事前確定するため、ClipLibraryView が一瞬見えてチラつく問題を回避。

マイルーティン行タップ挙動（596896d）

旧実装では行タップが即撮影（.shootingSequential）だったが、意図不一致のため .clipLibrary に変更。

  • 行タップ（通常）ClipLibraryView を開く（startMode: .clipLibrary）
  • 行末 ⋯ ボタンタップiOS 標準 .contextMenu フローティングメニューを表示（長押し contextMenu と同一メニュー。v1.1.0 で常設化）
  • 長押し → contextMenuiOS 標準 .contextMenu フローティングメニューを表示（v1.4 で confirmationDialog から変更）。⋯ ボタンと同一メニューを両経路で提供
  • 「すべて見る」ボタンv1.0 では未実装（アクション空）のため削除。全件表示に変更（prefix(4) 廃止）

ルーティンメニュー（contextMenu・v1.4 変更）

マイルーティンカード行末の ellipsis（⋯）ボタンタップ、または長押しで iOS 標準のフローティングメニューを表示。どちらの経路でも同一メニューを表示する（v1.1.0 で ⋯ ボタンを常設化し発見可能性を改善）。v1.4 以前は confirmationDialog（画面下部シート）だったが、.contextMenu に変更。メニュー項目は DesignsHomeView.myRoutinesListSection に集約（DesignsMyRoutineCard の onLongPressGesture / onMenuTap は削除済み）。

  1. 全ステップ続けて撮影 → ShootingView（startMode: .shootingSequential）
  2. クリップライブラリを開く → ClipLibraryView（startMode: .clipLibrary）
  3. 編集する → RoutineEditView
  4. 複製する → duplicate(routine:) で実行
  5. 削除する（role: .destructive）→ 削除確認 alert → deleteRoutine() で物理ファイル削除 + cascade delete

課金壁とサブスク連携

  • 無料ルーティン上限2件まで（maxFreeRoutines = 2）
  • 3件目作成時PaywallView 自動起動（trigger: .routineLimit）
  • 複製時の課金壁無料上限到達中は Paywall 起動（trigger: .routineLimit）して複製拒否
  • Quick Start タップ時の課金壁同上（trigger: .routineLimit）
  • 保存メーター Paywall最近の動画のストレージ上限時（trigger: .storageLimit）
  • Pro バッジ（無料時）手動タップtrigger: .manual

  • Pro バッジPro時は非インタラクティブ（情報のみ）、無料時は Paywall 起動ボタン

削除処理（deleteRoutine）

ルーティン削除時の物理ファイル + DBレコードの安全な削除順序:

  1. Array(routine.clips) スナップショット化（並行更新防止）
  2. 各 Clip の物理ファイル削除（ClipStorageService.deleteClip）
  3. Array(routine.videos) スナップショット化
  4. 各 Video の物理ファイル削除（VideoStorageService.deleteVideo）
  5. modelContext.delete(routine) で SwiftData cascade 削除

削除確認 alert で紐づくクリップ数・動画数をユーザーに提示。

通知ルーティング

リマインダー通知タップで起動された場合、NotificationRouter.shared.pendingRoutineId を監視し、該当ルーティンの撮影画面（ShootingView）を fullScreenCover で起動する。

  • トリガー.onAppear および .onChange(of: pendingRoutineId)
  • 二重発火防止oldValue != nil ガードで nil リセット時の不要再発火を防止
  • 処理後pendingRoutineId = nil でクリア

Hero streak メーター（v1.0 / 2026-06-10）

routines.flatMap(\$0.clips) を横断し、「今日 or 昨日に撮影記録あれば streak 継続、なければ 0 リセット」のルールで連続日数を算出してヒーロー上部に表示する。

Routine モデル拡張

  • lastShotAtcomputed property。routine.clips.map(\.recordedAt).max()
  • shotTodaycomputed property。Calendar.current.isDateInToday(lastShotAt)（クリップを1本以上撮影した日か。マイルーティンカードの「今日 ✓」緑バッジ判定に使用。Hero 判定には不使用）
  • hasCompletedVideoTodaycomputed property。routine.videos.contains { Calendar.current.isDateInToday($0.createdAt) }（v1.2.0 追加。完成動画 = Video レコードが今日 createdAt のものがあるか。Hero「今日はもう1本つくれています」判定に使用。クリップ撮影のみで動画未生成の状態では true にならない）
  • currentStreakcomputed property。単一 Routine の連続日数
  • computeStreak(from:) static複数 Routine 横断 helper。DesignsHomeView の overallStreak / totalVideoCount / overallHasCompletedVideoToday から呼び出す

表示パターン（3分岐）

条件バッジメインコピーサブコピー
記録ゼロ（overallStreak == 0 かつ totalVideoCount == 0）なし今日もルーティンを記録しよう5分だけでも。
overallHasCompletedVideoToday == false（今日完成動画なし・記録あり）🔥 N日連続 / 📹 累計 N 本（あれば）次の動画を1本つくろう—
overallHasCompletedVideoToday == true（今日完成動画あり）🔥 N日連続 / 📹 累計 N 本（あれば）今日はもう1本つくれています—

Hero コピー「週2本制作ツール」表現への統一（d59a710）

プロダクトポジショニングを「毎日続ける習慣アプリ」から「週2本つくる制作ツール」に統一するため、Hero メインコピーの分岐条件とコピー文言を変更した。

  • 分岐条件の変更（d59a710）旧条件: overallStreak >= 1（連続日数があるか）→ 新条件: overallHasShotToday（今日クリップを撮影済みか）。コピーが「作り続けること」ではなく「今日1本つくる」動機に直接紐づくよう変更
  • 分岐条件の再変更（v1.2.0 build68・F2変更）overallHasShotToday（クリップ撮影ベース）→ overallHasCompletedVideoToday（完成動画ベース）に変更。クリップを撮っただけで動画を生成していない状態では「今日はもう1本つくれています」を表示しない。Hero の「つくれています」を書き出し済み Video レコード（Video.createdAt が今日）の達成と一致させる
  • 旧コピー（streak ≥ 1 時）「連続◯日、続いてます」（習慣アプリ文脈）
  • 新コピー（今日完成動画なし）「次の動画を1本つくろう」（制作ツール文脈・行動喚起）
  • 新コピー（今日完成動画あり）「今日はもう1本つくれています」（達成感・追加制作への誘導）
  • 連続日数バッジ・shotToday バッジ🔥 N日連続 / 📹 累計 N 本 バッジ（streakBadge）は補助情報として継続表示（影響なし）。マイルーティンカードの「今日 ✓」緑バッジは shotToday（クリップ判定）のまま据え置き（完成動画とは意味が異なるため）

ステップ名も日本語化（wake up→起きる, skincare→スキンケア, go out→出発, warm up→ウォームアップ ほか）。raw value は永続化キー・URL scheme パスとして不変。

URL スキーム消費フロー

  • スキームroutinereel://preset/<rawValue>（project.yml + Info.plist の CFBundleURLTypes に登録）
  • 受け口RoutineReelApp.body.onOpenURL でパースして AppSettings.pendingPresetRawValue に格納
  • 消費DesignsHomeView の .onAppear および .onChange(of: settings.pendingPresetRawValue) で consumePendingPresetIfAny 実行。同名 Routine が既に存在すれば再利用、無ければ新規作成し activeSession を起動（fresh start / 既存セッション 両対応）

マイクロアニメーション

  • Hero フェードインopacity 0→1 + offset 6→0 を easeOut 0.45秒、感情コピーは 0.15秒遅延、CTA は 0.25秒遅延の spring
  • Quick Start カードappear時 spring(0.55, 0.85) でフェード+スライドアップ
  • 中央＋ボタン静的ハロー（pink 0.15）+ 押下時 scale 0.90 spring
  • 「次に撮影」CTA押下時 scale 0.97 spring、ハロー・脈動なし

常時アニメーションは意図的に最小化（GPT評価「アニメーション頑張った感」回避のため）。

v1.4 で FeatureFlags.swift を削除し、DesignsHomeView を唯一のホーム画面として固定した。旧 HomeView への切替トグル（SettingsView「β機能」セクション）は廃止済み。ロールバックが必要な場合はコード修正・再申請が必要。

Quick Start チップの表示ロジック（692882f）

  • 新規ユーザー（routines.isEmpty）Quick Start チップを表示（初回の撮影開始導線として機能）

Quick Start タップの動線変更履歴

プリセットタップ後の動線は v1.3 以降で 2 度変更されている。

部分撮影中カード（v1.2）

先頭ルーティンが部分撮影中（isPartiallyShot == true）の場合、通常の「次に撮影」Hero CTA と排他表示にし、代わりに「撮影の続き」カードを Hero セクション内に表示する。

  • 表示条件routines.first?.isPartiallyShot == true（先頭ルーティンが部分撮影中の場合のみ）
  • カード内容ルーティン名 + 「残り X ステップ」を表示
  • タップ動作startMode: .resumeShooting で ShootingView を起動。skipStepsWithExistingClips モードで未撮影ステップのみを連続撮影
  • 排他表示このカードが表示されている間は「次に撮影」CTA は非表示（重複を避ける）
  • 判定ソースRoutine.isPartiallyShot（モデルに集約された computed property）
  • 関連続きから撮る導線の機能仕様書

  • 変更前中央＋ボタンが offset(y: -18) でバナー領域に重なり、AdMob 誤タップリスクがあった
  • 変更後バナー直下に 24pt 非インタラクティブ離隔スペーサーを挿入（タッチイベントを受け取らない）

マイルーティン ellipsis ボタン常設化（v1.1.0）

編集・複製・削除などのルーティン操作が長押し contextMenu にしか存在せず、発見性が低かった。v1.1.0 でマイルーティン各行末に ellipsis（⋯）ボタンを常設し、ドリルダウンしなくても操作に気づける設計に変更した。

  • 変更前操作メニューは長押し contextMenu のみ。初見ユーザーは気づきにくかった
  • 変更後行末の ⋯ ボタンタップで contextMenu を即時表示。長押しも引き続き動作し、両経路で同一メニューを提供
  • 実装Button { } label: { Image(systemName: "ellipsis") } を各マイルーティン行 HStack の trailing に配置し、.contextMenu を呼び出す
