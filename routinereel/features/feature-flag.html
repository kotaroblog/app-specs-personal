FeatureFlag 運用

新UI投入時のリスク管理と即時ロールバック手段

目的

v1.0 リリース直前に大幅な UI 変更（DesignsHomeView）を投入するため、リリース後にクラッシュ・UX 不具合が発覚した場合に、 コード変更・再申請を経ずに旧UIへ即時切替できる経路を確保する。

実装方式

  • 実装ファイルApp/FeatureFlags.swift
  • 永続化方式@AppStorage(UserDefaults)
  • フラグキーuseDesignHome
  • デフォルト値true（新UI有効）
  • スコープv1.0 では DesignsHomeView の表示切替のみ

コード構造

// App/FeatureFlags.swift
enum FeatureFlags {
    static let useDesignHomeKey = "useDesignHome"
    static let useDesignHomeDefault = true

    static var useDesignHome: Bool {
        UserDefaults.standard.object(forKey: useDesignHomeKey) as? Bool
            ?? useDesignHomeDefault
    }
}

// App/RootView.swift
@AppStorage(FeatureFlags.useDesignHomeKey)
private var useDesignHome: Bool = FeatureFlags.useDesignHomeDefault

var body: some View {
    if hasCompletedOnboarding {
        if useDesignHome {
            DesignsHomeView()
        } else {
            HomeView()
        }
    } else {
        OnboardingView { ... }
    }
}

ユーザー操作経路

リリースビルドでも、ユーザー自身が以下の手順で新旧UIを切替えられる。

  1. アプリ起動
  2. 右上の歯車アイコンタップ → SettingsView
  3. 「β機能」セクションの「新しいホーム画面を使う」トグルを操作
  4. シートを閉じる → 次回起動時に切替が反映

運用シナリオ

シナリオ対応
リリース後にDesignsHomeViewでクラッシュ報告 SNS・サポートメールで「設定 → β機能 → 新しいホーム画面を使う を OFF」を案内。同時に修正版を準備
特定機種で UI 崩れ ユーザーに個別切替を案内。次バージョンで該当機種向け修正をリリース
v1.1 で新UIが安定 β機能セクションを削除し、旧 HomeView を完全に廃止
TestFlight 検証中に致命的バグ発見 FeatureFlags.useDesignHomeDefault を false に戻して再ビルド・提出

セキュリティ・プライバシー

  • UserDefaults に保存される値は Bool 1個のみで機密性なし
  • 外部送信されない（リモート切替ではなくローカル切替のみ）
  • App Privacy 申告への追加項目は不要

v1.1 以降のロードマップ

  • 新UIで重大バグ報告がゼロなら、v1.1リリースと同時に旧 HomeView を削除
  • 削除と同時に FeatureFlags.useDesignHome 関連コードと SettingsView の「β機能」セクションを削除
  • 残置期間の目安: v1.0 リリース後 1〜2 ヶ月
