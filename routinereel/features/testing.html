テスト基盤

ユニットテスト導入と、テスト経由で発見した本番バグの記録

導入の背景

v1.0 リリース直前まで RoutineReelTests/ ディレクトリ自体が存在せず、UITest はスクショ撮影専用の 1 メソッドのみという状態だった。Routine.computeStreak / SubscriptionManager.yearlySavingPercentage / Pro 解約 resolved パターンなどの計算ロジックがリグレッション保護無しで動いており、v1.1 以降のリファクタが律速する懸念があった。

ターゲット構成

  • ターゲット名RoutineReelTests（type: bundle.unit-test）
  • scheme 統合schemes.RoutineReel.test.targets に追加。xcodebuild test で UITest と並んで実行される
  • 依存RoutineReel 本体のみ（外部依存なし、純粋関数中心のテスト）

テスト経由で発見した本番バグ

SubscriptionManager.yearlySavingPercentage が **常に 0 を返していた** ことをユニットテストで検出。

原因

Swift の Decimal 型の / 演算子は内部で NSDecimalDivide(.plain) を scale=0 で呼ぶため、Decimal(1960) / Decimal(5760) が 0（整数除算的）になっていた。

影響

Paywall の年額カード「X% お得」バッジが「0% お得」と表示されていた可能性大（実機 / Firebase Analytics の paywall_shown 経由で要検証）。

修正

計算を Double 経由に変更:

let savingDouble = NSDecimalNumber(decimal: saving).doubleValue
let totalDouble = NSDecimalNumber(decimal: monthlyTotal).doubleValue
return Int((savingDouble / totalDouble) * 100)

同じ式をテストヘルパーにも反映し、5 ケースで挙動を固定化。

運用ルール

  • ロジックに変更があれば原則テスト追加（ViewModel・Service・Model の computed property）
  • テストファイルが存在するロジックを修正したら、必ずテスト同時更新（CI 化はリリース後）
  • UI 寄りのテストは UITest（スクショ撮影のみ）に集約し、純粋関数はユニットテストで担保する分業

build69 追加テストの観点

NextRecordingSelectorTests

  • 全ルーティン未撮影時は createdAt 最古のルーティンを選ぶ
  • 今日撮影済みのルーティンより未撮影を優先する
  • 部分撮影中（isPartiallyShot == true）のルーティンは選定対象外
  • 全ルーティンが部分撮影中の場合は nil を返す
  • lastShotAt が同じ場合は createdAt、さらに同じなら id 昇順で安定ソート

VideoCompletionTests

  • 同一 sessionId で2回 recordIfNeeded を呼んでも VideoCompletion が1件のみ
  • 異なる sessionId は別々に記録される
  • insertIfNeeded: 新規なら非 nil、既存なら nil を返す
  • in-memory ModelContainer のみで依存なくテスト可能

ClipLibraryCTATests

  • クリップ 0 件 → 「全ステップを撮影」を主 CTA とし副 CTA は nil
  • 一部撮影済み → 「残り N ステップを撮影」を主 CTA、「今あるクリップで動画を作る」を副 CTA
  • 全ステップ撮影済み → 「動画を作る」を主 CTA とし副 CTA は nil
  • N（unshotSteps.count）の計算が正しいこと

残課題

  • VideoCompositionService の動画合成ロジック（SceneLabelLayout 座標計算）にテスト未配置
  • CI（GitHub Actions）での自動実行は未設定。リリース後に Xcode Cloud / GitHub Actions の導入検討
