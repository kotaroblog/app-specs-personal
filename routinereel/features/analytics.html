Analytics 計測基盤（Firebase）

v1.0 リリース前に導入した KPI 計測の全貌（CHROMATIC / 脳活ピース で指摘済みの「計測基盤ゼロ」問題への対応）

導入の背景

  • 目的streak / Paywall / SNS シェア / 翌日通知 等のリテンション・収益施策の効果を測定可能にする
  • 背景CHROMATIC / 脳活ピースの GPT 批評（2026-05-20）で「KPI 計測基盤ゼロ」を指摘されており、RoutineReel リリース前に解消する必要があった
  • 採用ツールFirebase Analytics（同一 Google アカウントで他アプリも管理、Crashlytics へ拡張可能）
  • Firebase プロジェクトroutinereel-prod
  • 送信なしスクショ撮影モード（AppEnvironment.isUISnapshot）、GoogleService-Info.plist 未配置時、FirebaseApp.configure() 未実行時

アーキテクチャ

Firebase SDK への依存を Services/AnalyticsService.swift 1 ファイルに集約する。将来 TelemetryDeck 等への差し替えに備えた抽象化。

初期化フロー（RoutineReelApp.init）

  1. スクショ撮影モード判定（AppEnvironment.isUISnapshot）
  2. GoogleService-Info.plist 存在確認
  3. 両方クリアした時のみ FirebaseApp.configure() 実行
  4. ※ plist が無い場合でもクラッシュさせない（開発初期 / TestFlight 配布前の安全策）

イベント送信の 2 段ガード

AnalyticsService.log(_:params:) 内で以下を順番に確認:

  1. AppEnvironment.isUISnapshot が true → 即 return（撮影中の本番データ汚染防止）
  2. FirebaseApp.app() != nil でない → 即 return（configure 未実行時のクラッシュ防止）
  3. 両方通過したら Analytics.logEvent + ローカル Logger にも記録

必須 8 イベント定義

初回判定の永続化

AppSettings.hasRecordedFirst を UserDefaults で永続化し、first_recording イベントが 1 ユーザー 1 回だけ送信されることを保証する。

Firebase Analytics 命名規約への準拠

  • イベント名: 40 文字以内（snake_case）、予約プレフィックス（firebase_ / google_ / ga_）回避
  • パラメータ名: 40 文字以内（snake_case）
  • パラメータ値（文字列）: 100 文字以内（UUID 36 文字、enum 値は 20 文字以内で余裕）

セキュリティ

  • GoogleService-Info.plist は .gitignore に登録、git 管理外
  • 送信パラメータに個人情報・機密データを含めない（routine_id は UUID で氏名等を含まない）
  • Bundle ID は本番版（com.happyboy1002.RoutineReel）と一致

v1.2 追加: 9 イベント（失敗系・ファネル計測）

v1.0 の成功系 8 イベントだけでは「なぜ保存に至らないか」が見えないため、撮影中断・書き出し失敗・Paywall 離脱・購入失敗・ATT 結果を計測して離脱ポイントを特定する。9月末グロース判定の材料として活用する。

イベント名発火タイミングパラメータ用途
capture_startedセッション内の初回録画開始時（1セッション1回）total_steps: Int撮影完遂率算出の分母
capture_interrupted中断ダイアログで「残して終了」または「破棄して終了」を選択recorded_clips: Int, total_steps: Int何ステップ目で離脱するかの可視化
capture_completed全ステップの撮影セッションが正常完了recorded_clips: Int, total_steps: Int撮影完遂率算出の分子
compose_started動画合成処理を開始したとき（ClipGenerateView から）clip_count: Int合成成功率算出の分母
compose_completed動画合成処理が正常完了（保存完了とは別）duration_sec: Int端末別待ち時間の把握（長時間化は離脱要因）
export_failed書き出し・保存フローでの失敗（キャンセルを含む）stage: compose/filter/save_app/save_photo
reason: memory_pressure/file_missing/composition_error/filter_error/storage_error/save_failed/photo_permission_denied/incompatible_format/cancelled の直積		失敗カテゴリ別の発生率計測
paywall_dismissedPaywallView を購入せず閉じたときtrigger（paywall_shown と同一列挙）trigger 別の離脱率（paywall_shown - paywall_converted の差分と突合）
purchase_failed購入フローが成功以外で終わったときreason: cancelled/pending/verification_failed/store_errorキャンセル率が高い場合は価格・訴求の見直し材料
att_resultATT プロンプト表示後（1回のみ）authorized: Bool広告収益予測と ATT 表示タイミング再検討の判断材料

ファネル定義

ファネルイベント列失敗の集約先
撮影完遂capture_started → capture_completedcapture_interrupted
動画書き出しcompose_started → compose_completed → video_saved（既存）export_failed（全ステージ共通）
Paywall 転換paywall_shown → paywall_converted（既存）paywall_dismissed
購入フロー（ストア画面表示）→ paywall_convertedpurchase_failed

残課題（リリース後対応）

  • PaywallView の trigger 引数化: 現状 .manual 固定。各起動箇所（routine 上限 / storage 上限 / Pro フィルタ・フォントタップ）で trigger を渡すように PaywallView.init を変更
  • Firebase Console（routinereel-prod）でイベント受信確認: 実機 or Xcode 起動シミュレータで DebugView に 8 イベント表示確認
  • Crashlytics 統合: 同 plist 流用可、FirebaseCrashlytics product を SwiftPM 追加するだけ
  • D7 / D30 リテンション、Pro 転換率の初期ベースライン取得: リリース後 1 ヶ月で v1.1 値上げ判定に使う
