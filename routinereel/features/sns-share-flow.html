SNS シェアフロー

ExportComplete から TikTok/Reels 1tap 直起動 + 翌日復帰通知

1. 概要

「動画作成 → 投稿」までの心理的摩擦を最小化するため、ExportCompleteView から TikTok / Instagram Reels を 1tap で直起動する。シェア成功の瞬間を満足ピークと捉え、同タイミングで レビュー依頼と翌日復帰通知を仕込むことで、継続率とアプリ評価の双方を底上げする。

  • 対象画面Views/Export/ExportCompleteView.swift
  • 関連サービスReminderService / ReviewRequestService / InterstitialAdManager
  • 関連 commit8e00351 / 9322fd7

2. ボタン構成

視線の落ちる順に「主役 → 副役 → 脇役 → 完了」の 4 ボタンを縦に配置。主役以外はトーンを落とし、主役の TikTok を最も目立たせる。

役割内容
主役「TikTok を開いて投稿」（ピンクオレンジグラデ）
副役「Instagram Reels を開いて投稿」（border）
脇役「その他のアプリでシェア」（テキストリンク）
完了「完了」（最下部、InterstitialAdManager 経由でホーム復帰）

2.1 投稿手順インラインガイド（v1.1）

TikTok / Instagram Reels の各ボタンはアプリを開くだけで動画を自動転送しない。初見ユーザーが「ボタンを押したのに投稿できない」と混乱しないよう、ExportCompleteView にインラインガイドを追加した。

  • 動画の扱いエクスポート完了時点でカメラロールに保存済み
  • ガイド文言「動画はカメラロールに保存済みです。アプリを開いたら、ライブラリから動画を選んで投稿してください。」
  • 表示位置TikTok / Reels ボタンの直下（テキストラベルとして配置）
  • ボタンラベル変更前「TikTok で投稿」/「Instagram Reels で投稿」（自動投稿と誤解させる表現）
  • ボタンラベル変更後「TikTok を開いて投稿」/「Instagram Reels を開いて投稿」（アプリを開く動作であることを明示）

3. URL スキーム順序

各 SNS で公式アプリ未インストール時にも自然にフォールバックする多段構成。

遷移先1段目2段目3段目（fallback）
TikToktiktok://snssdk1233://ShareSheet
Instagram Reelsinstagram://libraryinstagram://ShareSheet

iOS 14 以降、canOpenURL 検知には Info.plist の LSApplicationQueriesSchemes 登録が必要。以下 4 種を必ず登録する。

  • tiktok
  • snssdk1233
  • instagram
  • instagram-stories

4. ハッシュタグ先回りコピー

投稿アプリ側でユーザーに手入力させない設計。ExportCompleteView 到達時点でハッシュタグを 自動でクリップボードに格納し、見出しテキストで「コピー済み」事実を明示する。

  • 発火タイミングonAppear で copyHashtagsToPasteboard()
  • 見出し文言「投稿時に使うハッシュタグ（コピー済み・投稿画面で長押し→ペースト）」
  • 冪等性各シェアボタンタップ時にも再コピー（クリップボードが上書きされていても確実に直前状態にする）

5. notifyShareTriggered の発火条件

シェア成立 = 投稿アプリへ確実に遷移できた、あるいは ShareSheet で完了したケースのみカウントする。 キャンセル時に発火させると過大評価になるため、以下条件を厳格に分ける。

導線発火条件label
TikTokUIApplication.open() success コールバック内「TikTok」
Instagram ReelsUIApplication.open() success コールバック内「Reels」
ShareSheetcompletionWithItemsHandler の completed=true「ShareSheet」
キャンセル発火しない―

6. ShareSheet ラッパー

既存の ShareSheet SwiftUI ラッパーに onCompletion クロージャを追加。 UIActivityViewController.completionWithItemsHandler を bridge し、 上位 View で完了 / キャンセルを区別できるようにする。 

struct ShareSheet: UIViewControllerRepresentable {
    let items: [Any]
    var onCompletion: ((Bool) -> Void)? = nil

    func makeUIViewController(context: Context) -> UIActivityViewController {
        let vc = UIActivityViewController(activityItems: items, applicationActivities: nil)
        vc.completionWithItemsHandler = { _, completed, _, _ in
            onCompletion?(completed)
        }
        return vc
    }
}

7. レビュー依頼

シェア完了は感情ピークとして最適。ReviewRequestService.onSharedToSNS() を発火条件と同タイミングで呼び出す。

  • 30 日ガードを他導線（撮影回数ベース等）と共用し、同一ユーザーへの過剰露出を防止
  • iOS の SKStoreReviewController.requestReview はシステムが年間 3 回までしか出さない点を前提に組む

8. 翌日復帰通知

投稿直後の満足感を翌日に持ち越し、「もう 1 本撮る」習慣化を促す。

  • メソッドReminderService.scheduleSharePromptNotification()
  • 発火時刻翌日 10:00
  • 識別子 prefixshare-prompt
  • 重複防止pending + delivered 両方を事前クリーンアップしてから登録
  • 許可ステータス.authorized または .provisional を許可とみなす
// 重複防止の擬似コード
let center = UNUserNotificationCenter.current()
center.removePendingNotificationRequests(withIdentifiers: [sharePromptId])
center.removeDeliveredNotifications(withIdentifiers: [sharePromptId])
// その後 add(request) で登録

9. URL スキーム経由のディープリンク（#6 連動）

外部メディア（TikTok bio / Qiita / note）から特定プリセットへ直送する集客導線。 SNS シェアフローと対をなすインバウンド経路として運用する。

  • URL 形式routinereel://preset/<rawValue>
  • rawValue 例morning / night / workout / study
  • 処理経路RoutineReelApp.onOpenURL → AppSettings.pendingPresetRawValue → DesignsHomeView.consumePendingPresetIfAny
  • 既存ルーティン同名再利用同名 Routine があれば再利用、なければ新規作成して即 activeSession へ
  • 起動状態の両対応fresh start（onAppear）・既存セッション（onChange）の両方をフック
