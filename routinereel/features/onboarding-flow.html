オンボーディングフロー

初回起動 → 通知許可 → ATT 遅延発火のリテンション設計

概要

通知許可ダイアログを大半のユーザーに表示してリテンション基盤を構築する。 iOS の ATT（App Tracking Transparency）ダイアログとの被りを避けるため、ATT は オンボーディング完了後に遅延発火させる。通知許可ダイアログを先に表示し、続けて ATT を表示することで、両ダイアログの許可率を高める設計とした。

ページ構成

Page名前内容
0 welcome アプリ名・キャッチコピー・「はじめる」ボタン（旧: "get started"・596896d で日本語化）
1 features 「連続撮影」/「クリップライブラリ」/「自動編集」の3機能紹介（旧: "continuous" / "clip library" / "auto-edit"・596896d で日本語化）。「次へ」ボタン（旧: "next"）
2 preset プリセット選択（朝5分 / 夜の整い / 筋トレ10分 / 集中45分 /「あとで」（旧: "skip for now"・596896d で日本語化））
3 reminder 朝7:00 / 夜21:00 /「あとで」の3択。プリセット選択時のみ表示、「あとで」経路はバイパス

selectReminder の挙動

  • pendingRoutine nil 時通知許可ダイアログを温存して早期 return（ゴースト Routine 防止）
  • granted = trueRoutine に時刻設定 + scheduleReminder を呼出
  • granted = falsereminderEnabled は true のまま保持。後で iOS 設定で許可された場合、RootView.onAppear の refreshAllReminders で自動的にスケジュールされる

TabView の swipe 無効化

オンボーディング画面の TabView に DragGesture() を被せて swipe を無効化している。 これにより、ユーザーが Page 2（preset）を「あとで」操作以外で飛ばして Page 3（reminder）に到達することを防ぎ、 pendingRoutine が nil のまま reminder ページに到達する経路を塞いでいる。

ATT 遅延発火（RootView 移管）

ATT ダイアログは OnboardingView ではなく RootView で発火する。これにより通知許可ダイアログとの被りを解消し、 両方の許可率を引き上げる。

  • 判定基準ATTrackingManager.trackingAuthorizationStatus ベースで毎回判定（永続化フラグは使わない）
  • 重複防止@State hasLoadedAdsInSession でセッション内重複ロードを防止。アプリ kill 後の再起動では再判定される
  • 発火トリガhasCompletedOnboarding の onChange と onAppear の両方で発火（オンボ完了直後・既存ユーザーの起動時の両ケース対応）

リカバリ経路

初回ダイアログで通知を拒否したユーザーが、後から iOS 設定で許可した場合のリカバリ経路を用意している。

  • RootView.onAppear で refreshAllReminders(routines:) を呼出
  • 通知拒否後にユーザーが iOS 設定で許可 → 次回起動時に、塩漬けになっていた reminderEnabled = true の Routine が自動的にスケジュールされる
  • これにより「拒否したユーザーが後から許可した」という典型的なケースをコード変更なしで救える

セキュリティ・プライバシー

  • 通知許可・ATT 許可状態は iOS 側で管理。アプリ側で保持するのは Routine の reminderEnabled / reminderTime のみ
  • 外部送信は行わない（ローカル通知のみ）
  • ATT 拒否時は AdMob を non-personalized モードで初期化（広告は表示される）
