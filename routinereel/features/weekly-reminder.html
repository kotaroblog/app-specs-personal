週次リマインダー（曜日指定）

リマインダーを鳴らす曜日をルーティンごとに指定できる機能。v1.2 で追加。iOS pending 通知上限（64件）を考慮した安全な登録設計が特徴。

機能ID: weekly-reminder v1.2

概要

v1.1.0 以前のリマインダーは「毎日特定時刻に通知」の固定仕様だった。v1.2 で Routine.reminderWeekdays: [Int]（1=日曜〜7=土曜）フィールドを追加し、ルーティンごとに通知曜日を自由に選択できるようにした。

ユーザー価値: 平日だけ撮影するルーティン（例: 筋トレ10分）や週末専用ルーティン（例: 夜の整いルーティン）など、曜日を限定した習慣化が可能になる。

関連画面

  • ルーティン編集（RoutineEditView） — 曜日選択チップ UI の配置先
  • ホーム画面 — 通知タップ → 撮影画面へのルーティング先

データモデル変更

  • 追加フィールドRoutine.reminderWeekdays: [Int]（SwiftData @Model に追加）
  • 値の意味Calendar.weekday 準拠: 1=日曜, 2=月曜, 3=火曜, 4=水曜, 5=木曜, 6=金曜, 7=土曜
  • デフォルト値[1, 2, 3, 4, 5, 6, 7]（全曜日 = 従来の「毎日」と同じ挙動。既存ユーザーのマイグレーション安全策）
  • 空配列の扱いUI では空にさせない（最後の1曜日はタップ不可）。ReminderService は空の場合を通知なし扱いにし、cancelReminder を実行してフォールバック

UI 仕様（RoutineEditReminderSection）

  • 表示位置リマインダー Toggle の下・DatePicker の下に曜日選択チップを配置（Toggle 有効時のみ展開）
  • チップ並び順月・火・水・木・金・土・日（月曜始まり表示。内部の weekday 値は 2〜7,1 の順で管理）
  • 選択状態accent orange 背景 + 白テキスト
  • 未選択状態designsCard 背景 + charcoal テキスト
  • 最後の1曜日ガードチップタップ後、選択済みが0曜日になる操作を阻止。代わりに「最低 1 曜日は必要です」テキストを 2 秒間表示（Timer で非表示に戻す）。チップの選択状態は変化させない

処理フロー

sequenceDiagram participant U as User participant V as RoutineEditView participant RS as ReminderService participant UN as UNUserNotificationCenter U->>V: 曜日チップをタップ V->>V: reminderWeekdays を更新（最後の1曜日ガード） U->>V: 保存ボタンタップ V->>RS: scheduleReminder(for: routine) RS->>UN: pendingNotificationRequests() で現在の件数を確認 alt 上限超過（otherCount + needed > 60） RS-->>V: .limitReached V-->>U: 「通知の上限に達しました」アラート（前の設定を維持） else 全曜日（weekdays.count == 7） RS->>UN: 毎日繰り返しの単一トリガーを登録（1件） RS-->>V: .scheduled else 一部曜日（weekdays.count < 7） loop 各選択曜日 RS->>UN: 曜日別 CalendarTrigger を登録（identifier: routine.reminder.UUID.wdN） end alt 全曜日登録成功 RS-->>V: .scheduled else 一部だけ成功 RS-->>V: .partiallyScheduled(succeeded:total:) V-->>U: 「一部の曜日（N/M 曜日）のみ登録できました」アラート else 全曜日失敗 RS-->>V: .notScheduled end end

ビジネスルール

  • デフォルトは全曜日選択 — 既存ユーザーがアップデートしても従来と同じ「毎日通知」として機能する
  • 全曜日選択時は pending 1 件に集約 — Pro ユーザーが多数のルーティンを持つ場合に pending 上限を節約する最適化
  • pending 予算は 60 件 — iOS の pending 上限 64 件に対して、SNS シェア促進通知（1件）とのレースの余裕を見て 60 件を上限とする
  • 上限超過時は既存通知を維持 — 新設定のために既存通知をキャンセルしてから失敗するという「最も気付きにくい失敗」を防ぐ。先に上限チェックを行い、超過する場合はキャンセルせず .limitReached を返す
  • 最後の1曜日は解除不可 — 空配列を防ぐ UI ガード。解除しようとすると 2 秒間ヒントを表示して操作を無効にする
  • 部分失敗（partiallyScheduled）もユーザーに伝える — 設定画面上の表示と実際の通知挙動の乖離を避けるため、一部登録に失敗した場合もアラートで知らせる
  • キャンセル時は旧・新両方の identifier を除去 — 毎日用（ベース identifier）と曜日別（.wd1〜.wd7 suffix）の両方を cancelReminder(for:) で一括削除する

外部連携

エラー処理

発生条件ScheduleOutcomeUI 対応
通知許可が拒否済み.notScheduledアラートなし（SettingsView の通知許可誘導に委ねる）
pending 上限（60件）超過.limitReached「通知の上限に達しました。以前の設定を維持します。」アラートを表示。ルーティン保存自体は成功させる
一部曜日の center.add() 失敗.partiallyScheduled(succeeded:total:)「一部の曜日（N/M 曜日）のみ登録できました。」アラートを表示

実装メモ

  • 実装ファイル（サービス）RoutineReel/Services/ReminderService.swift
  • 実装ファイル（モデル）RoutineReel/Models/Routine.swift（reminderWeekdays: [Int] 追加）
  • 実装ファイル（UI）RoutineReel/Views/Home/RoutineEditView.swift（RoutineEditReminderSection）
  • 上限チェックのタイミングキャンセル「前」に行う。先にキャンセルしてから上限で失敗すると、以前の設定が消えた状態になる（最も気付きにくい失敗パターン）
  • 起動時のリフレッシュReminderService.refreshAllReminders(routines:) でアプリ起動時に全ルーティンの通知を再登録。updatedAt 降順で優先順位をつけ、上限あふれ分はスキップ
  • pending 件数の計算「このルーティン自身の既存分」は「キャンセル後に空く枠」として除外して計算する（まだキャンセルしていないので pending には含まれている）
