続きから撮る導線（Resume Shooting）

撮影を途中でやめたルーティン（部分撮影中）を検知し、残りのステップだけを連続撮影できる導線。v1.2 で追加。撮影中断のコストを下げ、継続率向上を図る。

機能ID: resume-shooting v1.2

概要

v1.1.0 で中断時のクリップ保持が実現したが、「続きを撮りたい」ユーザーへの復帰導線が不明確だった。v1.2 では「部分撮影中」の状態を Routine.isPartiallyShot として明示的に定義し、ホーム画面・クリップライブラリ画面の両方から「残りのステップだけを続けて撮る」フローを1タップで起動できるようにした。

関連画面

  • ホーム画面（DesignsHomeView） — 「撮影の続き」カードを Hero セクションに表示
  • クリップライブラリ（ClipLibraryView） — 「残りのNステップを続けて撮影」主役 CTA に切り替え
  • 撮影画面（ShootingView） — skipStepsWithExistingClips モードで未撮影ステップのみ巡回

モデル変更（判定ソースの集約）

判定ロジックを複数画面で持つのを避けるため、Routine モデルに computed property として集約する（単一の判定ソース）。

computed property定義用途
unshotSteps: [Step]クリップが 0 件のステップ（表示順）。sortedSteps.filter { clips(for: $0).isEmpty }「続きから撮る」モードで撮影対象にするステップ一覧
isPartiallyShot: Bool!unshotSteps.isEmpty && unshotSteps.count < sortedSteps.count（一部撮影済み・一部未撮影）「撮影の続き」カード表示 / CTA 切り替えの判定

各画面での表示変化

ホーム画面（Hero セクション）

  • 表示条件先頭ルーティン（routines.first）の isPartiallyShot == true
  • カード表示内容ルーティン名 + 「残り X ステップ」（unshotSteps.count）
  • タップstartMode: .resumeShooting で ShootingView を起動
  • 排他表示「撮影の続き」カードが表示されている間は「次に撮影」CTA を非表示にし重複を避ける

クリップライブラリ画面（CTA セクション）

  • 表示条件当該ルーティンの isPartiallyShot == true
  • 主役 CTA「残りの N ステップを続けて撮影」（accent orange・大型ボタン）→ startMode: .resumeShooting
  • 副 CTA「最初から撮り直す」（secondary tone・小型テキストボタン）→ startMode: .shootingSequential
  • 変更前「全ステップを連続で撮影」CTA のみ（一択）

処理フロー

sequenceDiagram participant U as User participant H as ホーム/ClipLibrary participant VM as ShootingViewModel participant SV as ShootingView participant CLV as ClipLibraryView U->>H: 「撮影の続き」カード or 「残りのNステップを続けて撮影」タップ H->>VM: SessionPayload(routine, startMode: .resumeShooting) VM->>VM: skipStepsWithExistingClips = true VM->>VM: targetSteps = routine.unshotSteps（未撮影ステップのみ） VM->>SV: 起動（未撮影ステップを順に表示） loop 各未撮影ステップ U->>SV: 録画・保存 SV->>VM: persistClip(step:) end alt 全ステップ撮影完了 VM->>CLV: ClipGenerateView に遷移（最新クリップを自動選択済み） note right of CLV: 既存ステップ分クリップ込みで動画生成可能 else 途中で中断（クリップを残して終了） VM->>VM: 撮影済みクリップをライブラリに保持 SV-->>U: 撮影画面を閉じる else 途中で中断（破棄して終了） VM->>VM: セッション内の全クリップを削除（DB→ファイルの順） SV-->>U: 撮影画面を閉じる end

撮影完了後の遷移変更

v1.1.0 まで: 撮影完了 → PreviewView（プレビュー画面）に直接 push

v1.2 変更: 続きから撮るモードの完了 → ClipGenerateView（クリップ選択画面）に遷移し、最新クリップを自動選択済みの状態で表示。既存ステップ分のクリップも含めて動画生成できる。

  • 変更理由続き撮影では既存ステップのクリップも動画に含めたい。プレビュー直行では既存クリップが自動で含まれない
  • 自動選択今回の続き撮影で追加されたクリップが ClipGenerateView で自動選択済み状態になる（全ステップ完了扱いで全クリップが揃う場合は全件自動選択）

ビジネスルール

  • 部分撮影中の判定は常に最新 — Routine.isPartiallyShot は computed property のためスナップショットを持たない。クリップの削除・追加が即時反映される
  • 「最初から撮り直す」は既存クリップを削除しない — 新たな連続撮影を開始するが、既存クリップは明示的に削除しない（ユーザーがクリップライブラリで確認・削除できる）
  • 全ステップ未撮影 or 全完了の場合は従来どおり — isPartiallyShot == false の場合、「続きから撮る」UI は表示されず従来の「全ステップを連続で撮影」CTA を表示する
  • 判定対象はホームでは先頭ルーティンのみ — ホームの「撮影の続き」カードは先頭ルーティン（routines.first）に限定する。複数の部分撮影中ルーティンへの対応は今後の課題

外部連携

連携先用途備考
SwiftData（Routine / Step / Clip モデル）unshotSteps の計算・クリップの削除削除は DB 確定後にファイル削除の順序で参照整合性を保証
ClipStorageService破棄時のクリップファイル削除DB delete → save → fileDelete の順序

エラー処理

発生条件対応
続き撮影中に破棄を選択DB 確定後にファイル削除（参照整合性保証）。ClipStorageService.deleteClip を DB save 完了後に実行
unshotSteps が空（全ステップ撮影済み）での resumeShooting 起動起動元でガード済み（isPartiallyShot == false なら「続きから撮る」UI が非表示）。万一到達した場合は ShootingViewModel 側で graceful に全ステップ撮影モードにフォールバック

実装メモ

  • StartMode 追加CreateSessionView.StartMode 列挙に .resumeShooting を追加
  • ShootingViewModel フラグskipStepsWithExistingClips: Bool を追加。true のとき targetSteps を routine.unshotSteps に変更
  • 判定ソースのルールRoutine.unshotSteps / Routine.isPartiallyShot のみを使用。各画面が独自の判定ロジックを持たない
  • capture_interrupted / capture_completed 計測resumeShooting モードでも同じ Analytics イベントを送信する（total_steps は unshotSteps.count ではなく sortedSteps.count で統一）
