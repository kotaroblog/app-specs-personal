KPIイベントロギング

Phase 1 の効果測定と機能継続判断を支える裏側のロギング基盤。ユーザー体験への変更なし。

機能ID: FEAT-02 Phase 1

概要

Phase 1（DailyReview・6maxデフォルト化）の効果測定を目的とした、 イベントベースのロギング基盤。Phase 1 では os.log に出力し、 将来的に Firebase Analytics へ差し替えられる抽象レイヤーとして設計する。 ユーザーから見える体験への変更は一切ない。

関連画面

  • ホーム画面 (HOME-01)
  • CPU対戦セットアップ (SETUP-01)
  • DailyReview（今日の1問）(FEAT-01)

アーキテクチャ設計

EventLogger はシングルトンとして提供し、呼び出し側は AppEvent enum のケースを渡すだけで ロギングが完結する。バックエンドの差し替え（os.log → Firebase Analytics）は EventLogger.swift 内のみを変更すれば全体に反映される設計。

イベント定義表

イベント名発火条件パラメータ関連機能
daily_review_card_shown TODAY'S NOTE カードがアクティブ状態で表示された source: String (continuation / weakness / random) DailyReview
daily_review_started TODAY'S NOTE カードをタップしてクイズ画面に遷移した source: String DailyReview
daily_review_completed DailyReview の1問完了画面が表示された source: String, correct: Bool DailyReview
lesson_started_after_daily DailyReview 完了から10分以内に START LESSON を押した なし DailyReview + ホーム
cpu_match_started CPU対戦のマッチが開始した（対戦画面へ遷移） tableSize: Int (6 or 9) 6max デフォルト化
cpu_setup_shown CPU対戦セットアップ画面が表示された なし 6max デフォルト化
cpu_setup_started_match セットアップ画面から「対戦開始」ボタンを押した なし 6max デフォルト化

Setup 離脱率の算出設計

「離脱した」イベントを直接ログしない設計を採用する。 代わりに cpu_setup_shown と cpu_setup_started_match の差分から 離脱率を算出する。 

離脱率 = 1 − (cpu_setup_started_match の件数 / cpu_setup_shown の件数)

この方式の利点: 「戻るボタンを押した」「アプリをバックグラウンドにした」など 複数の離脱経路を区別せず、セットアップ完了率として一元管理できる。

実装仕様

// Services/EventLogger.swift

enum AppEvent {
    // DailyReview
    case dailyReviewCardShown(source: DailyReviewSource)
    case dailyReviewStarted(source: DailyReviewSource)
    case dailyReviewCompleted(source: DailyReviewSource, correct: Bool)
    case lessonStartedAfterDaily

    // CPU対戦
    case cpuMatchStarted(tableSize: Int)
    case cpuSetupShown
    case cpuSetupStartedMatch

    var name: String { /* イベント名を返す */ }
    var parameters: [String: Any] { /* パラメータ辞書を返す */ }
}

@MainActor
final class EventLogger {
    static let shared = EventLogger()
    private let logger = Logger(subsystem: "com.happyboy1002.PokerLesson", category: "KPI")

    func log(_ event: AppEvent) {
        logger.info("[\(event.name)] \(event.parameters as AnyObject)")
        // Phase 2: Analytics.logEvent(event.name, parameters: event.parameters)
    }
}

ビジネスルール

  • ログに個人を特定できる情報を含めない（ユーザーIDは含めない設計）
  • Phase 1 は os.log のみ。外部サーバーへのデータ送信はしない
  • Firebase Analytics 導入時はプライバシーポリシーの更新が必要
  • イベント名・パラメータキーは変更しない（将来の時系列比較のため）。廃止する場合は deprecated コメントを付け、新規イベントを追加する

KPI 観測ポイント（リリース後30日）

指標イベント目標値判断
DailyReview 開始率 daily_review_started / DAU 30% 以上 未達 → カード導線見直し
DailyReview 完了率 daily_review_completed / daily_review_started 70% 以上 未達 → 問題難易度・完了画面UX見直し
完了後 LESSON 進行率 lesson_started_after_daily / daily_review_completed 5% 以上 5% 以上でゲートウェイ機能有効と判断
6max 利用率 cpu_match_started の tableSize=6 の比率 60% 以上 未達 → デフォルト設定の認知が不足
Setup 離脱率 1 − (cpu_setup_started_match / cpu_setup_shown) 20〜30% 未満 超過 → セットアップ画面のUX見直し

実装メモ

  • Services/EventLogger.swift（新規）: AppEvent 列挙型と EventLogger.shared シングルトン
  • Views/Settings/EventLogDebugView.swift（新規・#if DEBUG）: 直近100件のイベントを設定→開発者ツールから閲覧
  • AppEvent.parameters は [String: Any]。Firebase Analytics 移行時にそのまま logEvent に渡せる形式にする
  • lesson_started_after_daily の判定: @AppStorage("daily_review_last_completed_at") と START LESSON 押下時刻の差分が 600秒（10分）以内かで判断
